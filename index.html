01 — Red Team · AI-assisted · OSCP path

Entorno profesional de pentesting con Kali Linux + Claude Code

No quería un asistente que "tirara comandos", sino un copiloto con criterio. Construí un framework de skills modulares —enumeration, exploitation, linux-privesc, oscp-reporting, anti_loop, cve_validation, decision_tree, workflow— con una knowledge base que actúa como cerebro externo: registra errores, heurísticas y lecciones de cada máquina.

  • Reactor (HTB): recon → CVE-2025-55182 (RCE no-auth en Next.js) → dump SQLite → hashcat → SSH → Node inspector + túnel SSH → root.
  • Silentium (HTB): cadena app + contenedor + host hasta root.
  • Insight: el fallo nunca fue técnico, fue de hipótesis inicial. Corregirla lo resolvió limpio.

02 — Cloud Native · Platform Engineering · PoC

Management plane multi-cluster con Rancher sobre EKS

Varias apps en Kubernetes repartidas en 56 cuentas AWS, cada cluster administrado distinto. Armé una PoC para gestionarlos de forma centralizada, estandarizando observabilidad y seguridad: un cluster de gestión (EKS + Rancher) desplegado con AWS CDK desde el cual se importan los workloads.

  • Gobierno por capas: Argo CD (GitOps), Prometheus + Grafana, Istio + Kiali (L7), Cilium + Hubble / eBPF (L3/L4).
  • Kyverno para políticas de admisión y Flagger para canary releases automáticos por métricas.
  • Postura auditada con CIS Benchmark (Sonobuoy + kube-bench).

Caso de impacto

De 8 a 56 cuentas AWS, gobernadas.

En un entorno cloud multi-cuenta, la infraestructura vivía mezclada entre 8 cuentas AWS, sin orden por ambiente y configurada a punta de clicks. Lideré la migración a una AWS Landing Zone con Control Tower y la adopción de GitOps — convirtiendo el caos en una plataforma gobernada y medible.

Antes
  • 8 cuentas con infraestructura mezclada, sin separación por ambiente.
  • Configuración manual por consola; recursos expuestos como buckets S3 públicos y usuarios IAM con full access.
  • Sin cumplimiento NIST / CIS, sin monitoreo ni detección de amenazas.
  • Redes con segmentos amplios y security groups all-traffic sin trazabilidad.
  • Troubleshooting manual: logs extraídos a mano de los clusters EKS y enviados por chat.
Después
  • 56 cuentas gobernadas, aisladas por proyecto y ambiente (dev / qa / prod).
  • Identidades centralizadas en AWS Identity Center con permisos de mínimo privilegio.
  • Postura de seguridad medible con Security Hub frente a NIST, CIS y AWS Best Practices.
  • Detección de actividad anómala con GuardDuty y respuesta automatizada ante incidentes.
  • Despliegues autónomos vía GitOps (ArgoCD): visibilidad y troubleshooting sin tocar los clusters.

El resultado: equipos de desarrollo autónomos, una superficie de ataque acotada y gobernanza verificable — la base sobre la que después se montó el management plane multi-cluster con Rancher.
